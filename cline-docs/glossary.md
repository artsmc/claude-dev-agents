# Glossary
Last Updated: 2026-03-09

Domain-specific terms used throughout the Claude Code Development Environment codebase.

---

## A

**Agent** - A specialized development persona with specific expertise (e.g., code reviewer, frontend developer, security auditor). Agents are defined in markdown files with standardized YAML frontmatter (name, model, tools) and invoked during task execution to provide context-appropriate responses.

**Agent Module** - An extracted deep-reference file for large agents, stored in `agents/modules/`. Modules contain specialized content (e.g., compliance guidance, penetration testing patterns) loaded on-demand only when the task requires them. Keeps core agent files under 20KB.

**Agent Context Caching** - System for reducing token usage through intelligent file caching across agent invocations. Uses SHA-256 hash-based invalidation and tracks cache hits/misses in PM-DB. Achieves 40-66% token savings in multi-invocation tasks.

**Agent Invocation** - A tracked execution of a specialized agent. Recorded in PM-DB with file reads, cache metrics, token usage estimates, and completion status. Enables session resumption and performance analytics.

**API Designer** - Specialized agent for contract-first API design. Creates OpenAPI specifications, defines three-tier architecture (route → service → external), and provides implementation contracts before coding begins. Operates in design mode only (read-only tools).

---

## C

**Cache Components** - Next.js 16+ feature for caching React components. The system includes migration tools for enabling this mode.

**Cache Hit** - When a requested file is found in the agent context cache, avoiding a full file read and saving tokens (~0 token cost for cached content).

**Cache Miss** - When a requested file is not in the agent context cache and must be read from disk, incurring full token cost. The file is then cached for future use.

**cline-docs** - The documentation hub directory containing system architecture, module responsibilities, glossary, and tech stack documentation.

**Confidence Level** - Self-checking system used by agents to assess their understanding before proceeding: High (proceed), Medium (proceed with caution, state assumptions), Low (STOP and ask for clarification). Part of the Confidence Protocol required in all agent definitions.

**CVE** - Common Vulnerabilities and Exposures. Standardized identifiers for known security vulnerabilities tracked by the OSV database and detected by security-quality-assess.

---

## D

**Detect Drift** - The process of identifying discrepancies between documentation and actual codebase. Uses `detect_drift.py` to find undocumented modules and technologies.

**Document Hub** - A four-file documentation system (systemArchitecture.md, keyPairResponsibility.md, glossary.md, techStack.md) that maintains synchronized codebase documentation.

---

## E

**Execute Mode** - Part of the start-phase system. Mode 2 provides five-part structured execution with automated quality gates between tasks.

**Extract** - The process of pulling domain-specific terms, comments, or metadata from code. Used by extract_glossary.py and other analysis tools.

---

## F

**File** - Generic term for source code files, configuration files, or documentation files. Tools operate on various file types including .ts, .tsx, .js, .jsx, .py, and .md files.

**FRD** - Functional Requirements Document. Part of the spec system for defining feature requirements.

**FRS** - Functional Requirements Specification. Detailed specification document generated by the spec-plan skill.

---

## G

**Generate** - The process of creating documentation, specifications, or reports automatically from codebase analysis. Common in documentation and specification tools.

**Glossary** - This file. A domain-specific term dictionary extracted from code comments and context. Maintained in cline-docs/glossary.md.

---

## H

**Hook** - Automated workflow enforcement triggered by specific events (e.g., task-complete, phase-complete). Hooks are markdown files that guide automated validation and quality gates.

**Hookify** - Plugin system for configuring and managing hooks. Includes rule engine for blocking operations based on conditions.

---

## K

**Knowledge Base** - The memory-bank system's six-file structure for storing project knowledge, active context, progress, architectural decisions, and learnings.

---

## M

**MCP** - Model Context Protocol. Next.js 16+ exposes MCP endpoints for runtime diagnostics and tool invocation.

**Model Routing** - Cost optimization strategy that assigns different LLM models to agents based on task complexity. Opus (6 agents) handles deep reasoning tasks; Sonnet (13 agents) handles implementation and review at 40-60% lower cost. Configured via the `model:` field in agent YAML frontmatter.

**Memory Bank** - Knowledge storage system with six core files (activeContext.md, progress.md, productContext.md, systemPatterns.md, architectureDecisions.md, learnings.md).

**Module** - A logical grouping of related functionality in the codebase. Examples: skills system, hooks system, agents system, tools layer.

**Module Drift** - Discrepancy between documented modules and actual code structure. Detected by detect_drift.py.

---

## P

**Phase** - A structured development period containing 5-7 tasks with automated quality enforcement. Managed by the start-phase system.

**Phase Plan** - An approved blueprint for executing a phase. Contains task definitions, dependencies, and success criteria. Stored in PM-DB before phase execution begins.

**Phase Run** - An actual execution instance of a phase plan. Tracks start/end times, assigned agent, status, and links to task runs.

**Plan Mode** - Part of the start-phase system. Mode 1 provides strategic planning with human approval before execution begins.

**PM-DB** - Project Management Database. SQLite-based system for tracking projects, phases, phase plans, phase runs, task runs, quality gates, and metrics. Uses v2 schema (migrated 2026-01-30).

**Plugin** - Extensible functionality for Claude Code. Located in plugins/ directory. Includes marketplaces and official plugins.

---

## O

**OSV API** - Open Source Vulnerability database API (https://api.osv.dev). Used by security-quality-assess to query known CVEs in project dependencies. Includes 24h local caching for performance and offline support.

**OWASP Top 10** - List of the 10 most critical web application security risks published by the Open Web Application Security Project. Security-quality-assess covers A01-A07 (85% coverage): Broken Access Control, Cryptographic Failures, Injection, Insecure Design, Security Misconfiguration, Vulnerable Components, and Authentication Failures.

---

## Q

**Quality Gate** - Automated enforcement of lint, build, and test checks between tasks. Results are stored in PM-DB's quality_gates table. Implemented by quality_gate.py and triggered by hooks.

---

## R

**Rule** - A conditional statement in the hookify system that determines whether to allow or block an operation. Rules are defined in hookify.*.local.md files.

**Rules** - Plural of rule. Collection of conditions used by the hookify rule engine to enforce workflow constraints.

---

## S

**Shadow Git Snapshot** - Automatic git branch checkpoint created before Write/Edit operations by the PreToolUse hook (`hooks/shadow-snapshot.sh`). Creates a `shadow/{timestamp}` branch pointing at the current state, enabling instant per-file rollback via `git checkout shadow/{timestamp} -- path/to/file`. Branches are auto-pruned after 24 hours.

**Security Assessment** - Automated vulnerability scanning process that analyzes Python and JavaScript/TypeScript code for OWASP Top 10 vulnerabilities. Performed by security-quality-assess skill using 6 specialized analyzers with zero external dependencies.

**Self-Checking Pattern** - Comprehensive quality mechanism used by enhanced agents including: Memory & Documentation Protocol, Confidence Level System (🟢🟡🔴), Pre-Execution Verification checklists, "When to Ask for Help" guidance, and Phased Quality Checklists. Applied to 6 agents as of 2026-02-09.

**Skill** - A reusable, composable workflow invoked via slash commands (e.g., /document-hub-initialize, /start-phase plan). Skills are self-contained markdown files with embedded instructions.

**SLOC** - Source Lines of Code. Metric tracked by sloc_tracker.py to measure code growth or reduction during phases.

**start-phase** - The flagship system providing comprehensive phase management with quality gates, hooks, and tools. Includes plan mode and execute mode.

**Suppression System** - False positive management system for security findings. Uses `.security-suppress.json` file to document accepted vulnerabilities with justification, expiration dates, and audit trail. Each suppression requires a reason and can be set to re-evaluate on code changes.

**Sync** - The process of updating documentation or knowledge to match the current codebase state. Used by memory-bank-sync and document-hub-update.

---

## T

**Task** - An individual unit of work within a phase. Tasks are defined in phase plans and executed as task runs. Quality gates are enforced between each task.

**Team Coordination** - Multi-agent orchestration pattern for parallel task execution. Uses TeamCreate to spawn a team, shared TaskList for coordination, and SendMessage for explicit agent-to-agent communication. Enables simultaneous work on different components by specialized agents.

**TeamCreate** - Tool for spawning a team of specialized agents with shared task list coordination. Creates team configuration file at `~/.claude/teams/{team-name}/config.json` and initializes shared task list in PM-DB. Team members can communicate via SendMessage and coordinate via TaskList.

**Token Savings** - Reduction in API token usage achieved through agent context caching. Measured as percentage reduction from baseline (40-66% measured in multi-invocation tasks). Cache hits incur ~0 token cost vs full file read cost on cache miss.

**Task Run** - Individual task execution record within a phase run. Tracks task-level progress, exit codes, and completion status in PM-DB.

**Tech Stack** - The collection of technologies, frameworks, and tools used in a project. Documented in cline-docs/techStack.md.

**Technical Writer** - Specialized agent for creating user-facing documentation including API docs, README files, tutorials, changelogs, and inline code documentation. Read-only agent that transforms complex technical concepts into clear, accessible documentation.

**Technology Drift** - Discrepancy between documented technologies and actual dependencies. Detected by detect_drift.py from package.json and requirements.txt files.

**Tool Restriction Profile** - Per-agent access control defining which Claude Code tools an agent can use. Declared in YAML frontmatter `tools:` field. Profiles: read-only (Read, Grep, Glob), write-no-shell (adds Write, Edit), full (adds Bash). Enforces principle of least privilege — review agents can't write, spec agents can't shell.

**Token Budget** - The maximum number of tokens (context window) available for a conversation. The system uses Claude Opus 4.6 with 1M context window.

**Tool** - A Python utility script providing specific functionality (validation, quality checks, analysis). All tools use zero dependencies (Python stdlib only).

---

## V

**Validation** - The process of checking structure, syntax, and completeness of documentation, phases, or tasks. Implemented by validate_*.py scripts.

---

## W

**Working Memory** - Session-scoped shared markdown file (`working-memory/{team-name}.md`) used for cross-agent context sharing within a team. Agents write decisions, discoveries, blockers, and handoff notes. Created by team lead on TeamCreate, deleted on TeamDelete. Distinct from Memory Bank (episodic/permanent) — working memory is temporary and dies with the team session.

**Workflow** - A sequence of automated steps triggered by events. Implemented through hooks and enforced by quality gates.

---

## Z

**Zero Dependencies** - Design principle requiring all Python tools to use only the standard library. Ensures portability and eliminates installation friction.
